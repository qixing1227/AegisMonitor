# AegisMonitor 需求规格说明书

## 1. 引言

### 1.1 编写目的

本文档用于明确 AegisMonitor 一体化监控平台的功能需求、非功能需求、接口需求、数据需求和运行环境要求，为后续概要设计、详细设计、编码、测试和课程答辩提供依据。

### 1.2 项目范围

系统由 Windows Agent、Spring Boot 后端服务、Vue 前端监控台、MySQL 数据库和 InfluxDB 时序数据库组成。Agent 负责采集主机与服务数据，后端负责认证、接收、存储、查询和告警，前端负责可视化展示和运维操作。

### 1.3 术语

| 术语 | 含义 |
| --- | --- |
| Agent | 部署在被监控主机上的采集程序 |
| Host | 被监控主机 |
| Metric | 指标数据，例如 CPU 使用率、内存使用率 |
| Service Instance | 被识别出的服务实例，例如 MySQL、Redis |
| Alert Rule | 告警规则 |
| Alert Event | 告警事件 |
| RBAC | 基于角色的访问控制 |
| ACK | 告警确认操作 |

## 2. 总体描述

### 2.1 系统架构概述

系统采用前后端分离和 Agent 主动上报模式。

数据流如下：

1. Agent 读取本地配置。
2. Agent 使用注册 Token 调用后端注册接口。
3. Agent 定时采集主机指标和服务识别结果。
4. 后端接收数据后，将资产、服务、告警等关系型数据写入 MySQL，将高频指标写入 InfluxDB。
5. 后端根据告警规则生成告警事件。
6. 前端通过 REST API 定时拉取主机、指标、服务和告警数据。
7. 运维人员在前端确认和关闭告警。

### 2.2 用户特征

- 超级管理员：熟悉系统配置、用户管理和 Agent 纳管。
- 运维工程师：熟悉主机资源、服务状态和告警处理。
- 研发工程师/业务方：关注应用运行状态，具有只读查看需求。

### 2.3 假设与约束

- MVP 仅支持 Windows Agent。
- 演示环境接入 2 台真实主机，并准备 3 台模拟主机。
- 指标数据写入 InfluxDB。
- 关系型数据写入 MySQL。
- 前端采用定时轮询方式刷新数据。
- 不实现生产级高可用、自动扩缩容和完整分布式追踪。

## 3. 功能需求

### FR-001 用户登录

系统应支持用户通过用户名和密码登录。

验收标准：

- 登录成功后返回 JWT。
- 登录失败时返回明确错误信息。
- 前端保存 JWT 并在后续请求中携带。

### FR-002 角色权限控制

系统应支持基于角色的访问控制。

验收标准：

- 超级管理员可访问用户管理和 Agent 管理页面。
- 运维工程师可访问监控、告警和规则页面。
- 研发只读角色写入设计文档，MVP 可不实现真实隔离。

### FR-003 Agent 注册

Agent 应使用注册 Token 向后端注册。

验收标准：

- Token 正确时，后端创建或更新主机记录。
- Token 错误时，后端拒绝注册。
- 注册成功后返回 Agent 标识或主机标识。

### FR-004 Agent 心跳

Agent 应定时向后端发送心跳。

验收标准：

- 心跳周期默认为 10 秒。
- 后端记录最近心跳时间。
- 超过离线阈值时生成 Agent 离线告警。

### FR-005 主机基础信息采集

Agent 应采集主机名、操作系统、系统版本、CPU 核心数、内存总量和启动时间。

验收标准：

- 后端可保存并查询主机基础信息。
- 前端主机详情页可展示基础信息。

### FR-006 CPU 指标采集

Agent 应采集总 CPU 使用率和每核 CPU 使用率。

验收标准：

- 指标默认每 5 秒采集一次。
- 指标写入 InfluxDB。
- 前端可展示最近 10 分钟 CPU 趋势图。

### FR-007 内存指标采集

Agent 应采集内存总量、已用、可用和使用率。

验收标准：

- 指标写入 InfluxDB。
- 前端可展示内存使用率和趋势图。

### FR-008 磁盘指标采集

Agent 应采集磁盘分区、总量、已用、可用和使用率。

验收标准：

- 前端可查看各分区磁盘使用情况。
- 磁盘使用率超过阈值时可触发告警。

### FR-009 网络指标采集

Agent 应采集网卡名称、发送字节数、接收字节数、发送速率和接收速率。

验收标准：

- 前端可展示网络收发趋势。
- 网络数据默认每 5 秒采集一次。

### FR-010 TCP 连接信息采集

Agent 应采集 TCP 连接数量和监听端口列表。

验收标准：

- 前端可查看 TCP 连接总数。
- 前端可查看监听端口列表。

### FR-011 服务识别

Agent 应识别 Java/Spring Boot、MySQL、Redis、Nginx、Node.js 等典型服务。

验收标准：

- 服务发现周期默认为 30 秒。
- 可识别服务名称、技术栈类型、PID、端口和运行状态。
- 服务实例信息写入 MySQL。

### FR-012 服务指标采集

Agent 应采集服务进程的 CPU、内存和连接数等轻量指标。

验收标准：

- 服务指标写入 InfluxDB。
- 前端服务组件页面可展示服务进程状态。

### FR-013 主机分组与标签

系统应支持轻量主机分组和标签。

验收标准：

- 主机可标记为生产环境、测试环境、演示环境等分组。
- 主机可带有 SpringBoot、MySQL、模拟数据等标签。
- 主机列表可按分组、状态或标签筛选。

### FR-014 阈值告警规则

系统应支持 CPU、内存、磁盘、Agent 离线和服务停止告警规则。

验收标准：

- 运维工程师可查看和修改阈值。
- 告警规则保存在 MySQL。
- 指标超过阈值时生成告警事件。

### FR-015 告警事件管理

系统应支持告警事件查询、确认和关闭。

验收标准：

- 告警事件包含级别、对象、内容、发生时间、状态。
- 运维工程师可执行 ACK 并填写备注。
- 运维工程师可关闭已处理告警。

### FR-016 服务拓扑展示

系统应展示轻量服务拓扑。

验收标准：

- 前端可展示 Nginx、Spring Boot、MySQL、Redis 等服务依赖关系。
- 拓扑可基于服务识别结果和演示配置生成。

### FR-017 模拟调用链展示

系统应展示模拟调用链信息。

验收标准：

- 点击调用关系可查看模拟耗时、状态、时间点。
- 文档说明生产环境可接入 OpenTelemetry。

### FR-018 演示数据生成

系统应支持生成模拟主机、指标、服务和告警数据。

验收标准：

- 可生成 3 台模拟主机。
- 可生成最近 30 分钟指标数据。
- 可生成典型服务实例和告警事件。

### FR-019 API 文档

系统应提供 Swagger UI / OpenAPI 文档。

验收标准：

- 后端启动后可访问接口文档。
- 登录、Agent、主机、指标、服务、告警接口应被文档化。

## 4. 非功能需求

### NFR-001 可用性

系统应能在课程演示环境中稳定启动和运行。数据库使用 Docker Compose 启动，后端、前端和 Agent 本地运行。

### NFR-002 可维护性

后端采用模块化单体结构，按认证、Agent、主机、指标、服务、告警、模拟数据等模块划分。Agent 按采集器拆分主机指标和服务发现逻辑。

### NFR-003 可扩展性

系统应在设计上支持后续扩展 Linux Agent、远程配置下发、时序数据降采样、OpenTelemetry 追踪和外部告警通知。

### NFR-004 安全性

用户请求使用 JWT 鉴权。Agent 上报使用 Agent Token 或 Agent Secret 认证。不同角色具有不同权限。MVP 实现超级管理员和运维工程师角色。

### NFR-005 性能

在 2 台真实主机和 3 台模拟主机场景下，系统应支持 5 秒级指标采集和前端刷新。前端查询最近 10 分钟指标时应能在可接受时间内返回。

### NFR-006 可测试性

系统核心逻辑应支持单元测试、接口测试、集成测试和异常测试。告警判断、权限判断、Agent 数据转换和指标上报应作为重点测试对象。

## 5. 外部接口需求

### 5.1 Agent 上报接口

- Agent 注册接口。
- Agent 心跳接口。
- 主机指标上报接口。
- 服务发现结果上报接口。
- 服务指标上报接口。

### 5.2 前端业务接口

- 登录接口。
- 主机列表接口。
- 主机详情接口。
- 指标查询接口。
- 服务组件接口。
- 服务拓扑接口。
- 告警规则接口。
- 告警事件接口。
- 用户管理接口。

### 5.3 数据库接口

- MySQL：通过 MyBatis-Plus 访问。
- InfluxDB：通过 InfluxDB Java Client 访问。

## 6. 运行环境

### 6.1 开发环境

- 操作系统：Windows。
- Agent：Python。
- 后端：Java + Spring Boot。
- 前端：Vue 3 + Vite。
- 数据库：MySQL + InfluxDB。
- 数据库启动：Docker Compose。

### 6.2 演示环境

- 2 台真实 Windows 主机。
- 3 台模拟主机数据。
- 网络连通。
- 后端、前端、数据库和 Agent 可在局域网内互通。

## 7. 数据需求

### 7.1 MySQL 数据

- 用户表。
- 角色表或用户角色字段。
- Agent 表。
- 主机资产表。
- 服务实例表。
- 告警规则表。
- 告警事件表。
- 系统配置表。

### 7.2 InfluxDB 数据

- host_cpu_metric。
- host_memory_metric。
- host_disk_metric。
- host_network_metric。
- host_tcp_metric。
- service_process_metric。

## 8. 需求优先级

| 优先级 | 范围 |
| --- | --- |
| P0 | Agent 注册、心跳、主机指标采集、后端接收、MySQL/InfluxDB 存储、前端主机展示 |
| P1 | 服务识别、告警规则、告警事件、权限登录、演示数据 |
| P2 | 服务拓扑、模拟调用链、用户管理、Agent 纳管页面 |
| P3 | Linux Agent、远程配置、WebSocket、真实通知通道、OpenTelemetry |
