# AegisMonitor 可行性分析

## 1. 项目概述

AegisMonitor 一体化监控平台面向课程设计场景，目标是实现一个可运行、可展示、可测试的监控平台原型。系统通过 Windows Agent 采集主机指标和服务信息，由 Spring Boot 后端接收、存储和处理数据，并通过 Vue 监控台展示主机状态、服务组件、告警事件和服务拓扑。

本项目采用课程设计级可运行演示系统定位，不建设生产级企业监控系统，但在架构上采用 MySQL + InfluxDB、Agent Token、RBAC、告警闭环等接近真实企业系统的设计思想。

## 2. 技术可行性

### 2.1 Agent 技术可行性

Agent 使用 Python 实现，MVP 仅支持 Windows。Python 可通过成熟库采集主机指标，包括 CPU、内存、磁盘、网络、TCP 连接和进程信息。服务识别可通过进程名、端口和命令行参数实现轻量规则匹配。

可行性判断：

- Windows 主机环境可控。
- 采集指标与课程题目要求一致。
- Python 实现速度快，便于调试。
- 服务识别先覆盖 Java/Spring Boot、MySQL、Redis、Nginx、Node.js，范围可控。

主要风险：

- 个别服务在演示电脑上未安装。
- Windows 权限可能影响部分 TCP 或进程信息采集。

应对策略：

- 允许真实服务识别和模拟服务数据并存。
- 对权限不足的指标做异常处理和降级展示。

### 2.2 后端技术可行性

后端采用 Spring Boot 模块化单体架构，使用 MyBatis-Plus 管理 MySQL 中的关系型数据，使用 InfluxDB Java Client 管理时序指标数据。Spring Boot 适合 REST API、权限认证、定时任务、告警判断和 Swagger/OpenAPI 文档生成。

可行性判断：

- Spring Boot 是成熟企业级后端框架。
- MyBatis-Plus 能快速完成 CRUD。
- InfluxDB 适合存储 5 秒级高频指标。
- 模块化单体避免微服务带来的部署复杂度。

主要风险：

- MySQL 和 InfluxDB 双数据库增加配置复杂度。
- InfluxDB 查询模型与 MySQL 不同，需要额外学习。

应对策略：

- 使用 Docker Compose 统一启动数据库。
- MVP 只做最近 10 分钟、最近 30 分钟等简单时间窗口查询。

### 2.3 前端技术可行性

前端采用 Vue 3 + Vite + Element Plus + ECharts。Element Plus 适合后台管理台，ECharts 适合 CPU、内存、磁盘、网络等指标图表展示。

可行性判断：

- 前端框架成熟，资料丰富。
- 管理台布局与监控平台场景匹配。
- 定时轮询即可满足演示，不需要 WebSocket。

主要风险：

- 图表数据结构与后端返回格式不匹配。
- 页面功能较多，开发量可能偏大。

应对策略：

- 先完成主机列表和主机详情，再做仪表盘和拓扑。
- 后端接口返回适配 ECharts 的标准结构。

## 3. 经济可行性

项目所需软件均可使用开源或免费工具：

- Python。
- Java / Spring Boot。
- Vue 3 / Vite。
- Element Plus。
- ECharts。
- MySQL。
- InfluxDB。
- Docker Compose。

硬件方面，课程题目要求 PC 机 2 台，项目计划使用 2 台真实 Windows 主机和 3 台模拟主机数据进行演示。无需购买额外服务器或商业软件。

可行性判断：

- 成本可控。
- 满足课程硬件要求。
- 不依赖收费云服务或商业平台。

## 4. 操作可行性

系统面向运维人员设计，主要操作包括登录、查看仪表盘、查看主机列表、查看主机详情、处理告警、查看服务拓扑等。前端采用后台管理台风格，交互清晰，符合运维人员日常使用习惯。

可行性判断：

- 用户角色清晰。
- 主要操作路径短。
- 告警 ACK 和关闭流程贴近真实运维场景。

主要风险：

- 答辩时现场网络或真实 Agent 不稳定。

应对策略：

- 提供模拟主机和模拟指标。
- 准备答辩演示脚本和兜底路线。

## 5. 时间可行性

项目采用“文档化瀑布 + 敏捷迭代实现”的混合过程模型。课程文档按可行性分析、项目计划、需求分析、概要设计、详细设计、编码、测试组织；实际开发按 Sprint 推进。

计划阶段：

- Sprint 0：需求与规划。
- Sprint 1：主链路打通。
- Sprint 2：监控展示。
- Sprint 3：告警与权限。
- Sprint 4：测试与答辩。

可行性判断：

- MVP 范围已明确。
- 不做生产级高可用、远程配置、自动升级和完整 Trace。
- 任务可拆分到 5 个展示角色。

主要风险：

- 功能范围扩张。
- 同时处理 Agent、后端、前端和文档导致工作量过大。

应对策略：

- 严格按 P0、P1、P2 优先级执行。
- 先保证最小可演示闭环，再补加分功能。

## 6. 安全可行性

系统设计用户 JWT 认证和 Agent Token 认证两套机制。用户侧通过 RBAC 区分超级管理员、运维工程师和研发只读角色。Agent 侧通过注册 Token 控制接入。

可行性判断：

- MVP 实现超级管理员和运维工程师两个角色。
- 研发只读角色写入设计文档作为扩展。
- 用户认证和 Agent 认证分离，便于解释企业级安全边界。

主要风险：

- 完整权限体系实现成本较高。

应对策略：

- MVP 只实现两类核心角色。
- 第三类只读角色作为扩展设计。

## 7. 风险分析

| 风险 | 概率 | 影响 | 应对策略 |
| --- | --- | --- | --- |
| 数据库服务启动失败 | 中 | 高 | 使用 Docker Compose，提前准备演示数据和截图 |
| Agent 无法采集部分指标 | 中 | 中 | 异常处理，缺失字段前端降级展示 |
| 第二台真实主机无法联网 | 中 | 中 | 同机运行第二配置或使用模拟主机 |
| 服务识别结果不足 | 中 | 中 | 使用模拟服务实例补足 5 类组件展示 |
| 告警触发不明显 | 中 | 中 | 支持调低阈值和预置告警 |
| 前端页面开发量过大 | 中 | 高 | 优先主机列表、主机详情、告警中心 |
| 文档后期补写压力大 | 高 | 高 | 从当前阶段持续维护文档 |

## 8. 可行性结论

综合技术、经济、操作、时间和安全因素，AegisMonitor 项目具备可行性。项目范围已被限定为课程设计级可运行演示系统，技术栈成熟，硬件要求可满足，成本较低，功能能够按 Sprint 逐步实现。

项目最大风险是范围过大和演示环境不稳定。通过 P0/P1/P2 优先级控制、模拟数据兜底和文档化过程管理，可以将风险控制在课程设计可接受范围内。
