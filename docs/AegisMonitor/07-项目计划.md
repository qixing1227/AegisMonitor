# AegisMonitor 项目计划

## 1. 项目目标

AegisMonitor 项目目标是在课程设计周期内完成一套一体化监控平台原型，实现 Agent 数据采集、后端接收存储、前端监控展示、阈值告警、告警处理闭环、服务识别和演示数据支持。

最终交付物包括：

- 可运行软件系统。
- 源代码。
- PRD。
- 需求规格说明书。
- 可行性分析。
- 项目计划。
- 概要设计说明书。
- 详细设计说明书。
- 测试说明书。
- 课程设计报告。
- 答辩演示脚本。
- 五人展示分工说明。

## 2. 过程模型

项目采用“文档化瀑布 + 敏捷迭代实现”的混合过程模型。

对课程文档：

1. 可行性分析。
2. 项目计划。
3. 需求分析。
4. 概要设计。
5. 详细设计。
6. 编码。
7. 测试。
8. 课程设计报告。

对实际开发：

1. Sprint 0：需求与准备。
2. Sprint 1：主链路打通。
3. Sprint 2：监控展示。
4. Sprint 3：告警与权限。
5. Sprint 4：拓扑、测试与答辩。

## 4. Sprint 计划

### 4.1 Sprint 0：需求与准备

目标：

- 完成需求澄清。
- 完成 PRD 和需求规格说明书。
- 完成任务拆解。
- 完成可行性分析和项目计划。
- 明确技术选型和工程边界。

主要任务：

- 建立文档体系。
- 确认 MVP 范围。
- 确认 P0/P1/P2/P3 优先级。
- 规划项目目录结构。
- 规划数据库和部署方式。

交付物：

- 项目决策记录。
- PRD。
- 需求规格说明书。
- 验收标准与测试范围。
- 五人展示分工说明。
- 可行性分析。
- 项目计划。
- 任务拆解。
- MVP 功能优先级清单。

### 4.2 Sprint 1：主链路打通

目标：

- Agent 能注册。
- Agent 能心跳。
- Agent 能采集主机指标。
- 后端能接收指标。
- 指标能写入 InfluxDB。
- 资产数据能写入 MySQL。

主要任务：

- 搭建后端基础项目。
- 搭建数据库 Docker Compose。
- 创建 MySQL 基础表。
- 实现 Agent 注册接口。
- 实现 Agent 心跳接口。
- 实现主机指标上报接口。
- 实现 Python Agent 配置读取。
- 实现主机基础指标采集。
- 实现指标上报。

交付物：

- 可启动数据库环境。
- 可启动后端服务。
- 可运行 Agent。
- 可查询主机和指标。

验收标准：

- Windows Agent 使用 Token 成功注册。
- 后端能记录心跳。
- InfluxDB 中能看到主机指标。
- MySQL 中能看到主机资产。

### 4.3 Sprint 2：监控展示

目标：

- 前端能展示监控数据。
- 主机列表和主机详情可用。
- 服务识别结果可展示。

主要任务：

- 搭建 Vue 3 前端项目。
- 实现基础布局。
- 实现主机列表页面。
- 实现主机详情页面。
- 实现 ECharts 指标曲线。
- 实现服务识别上报接口。
- 实现 Agent 服务识别。
- 实现服务组件页面。

交付物：

- 可登录或可访问的前端监控台。
- 主机列表。
- 主机详情。
- 服务组件页面。

验收标准：

- 前端能看到真实或模拟主机。
- 主机详情能展示 CPU、内存、磁盘、网络、TCP 指标。
- 服务组件页面能展示典型服务。

### 4.4 Sprint 3：告警与权限

目标：

- 系统具备基本企业权限模型。
- 系统能触发并处理告警。

主要任务：

- 实现登录接口。
- 实现 JWT。
- 实现 SYSTEM_ADMIN 和 OPS_ENGINEER 两类角色。
- 实现告警规则管理。
- 实现告警事件生成。
- 实现告警中心。
- 实现告警 ACK 和关闭。
- 实现 Agent 纳管页面的轻量版本。

交付物：

- 登录页面。
- 权限控制。
- 告警规则页面。
- 告警中心。
- ACK 与关闭功能。

验收标准：

- 未登录不能访问受保护接口。
- 运维工程师不能访问管理员专属功能。
- 调低阈值后能触发告警。
- 告警可确认和关闭。

### 4.5 Sprint 4：拓扑、测试与答辩

目标：

- 完成加分展示能力。
- 完成测试与文档收口。
- 准备答辩演示。

主要任务：

- 实现模拟数据生成。
- 实现服务拓扑接口。
- 实现服务拓扑页面。
- 实现模拟调用链详情。
- 编写单元测试。
- 编写接口测试。
- 编写集成测试和异常测试。
- 编写测试说明书。
- 编写课程设计报告。
- 编写答辩演示脚本。

交付物：

- 模拟主机和指标数据。
- 服务拓扑页面。
- 模拟调用链页面。
- 测试说明书。
- 课程设计报告。
- 答辩演示脚本。

验收标准：

- 演示环境有真实主机和模拟主机。
- 服务拓扑可展示。
- 测试文档覆盖核心链路和异常场景。
- 答辩脚本可完整执行。

## 5. 任务依赖

关键依赖关系：

1. 数据库 Docker Compose 先于后端数据访问。
2. 后端 Agent 注册接口先于 Agent 注册功能。
3. 后端指标接收接口先于 Agent 指标上报。
4. 主机查询接口先于前端主机列表和详情。
5. 登录和 JWT 先于前端权限菜单。
6. 告警规则先于告警事件生成。
7. 服务识别先于服务组件和服务拓扑。
8. MVP 主链路先于测试说明书和答辩脚本。

## 7. 进度控制策略

项目采用优先级驱动：

- 先完成 P0，保证系统能跑通。
- 再完成 P1，保证答辩展示完整。
- 最后完成 P2，增加专业度和亮点。
- P3 只写入设计展望。

每个 Sprint 结束时进行一次检查：

- 当前功能是否可运行。
- 文档是否同步更新。
- 是否产生了新的风险。
- 是否需要削减 P2 功能保证 P0/P1。

## 8. 质量保证策略

质量保证包括：

- 需求阶段明确验收标准。
- 编码阶段按模块拆分。
- 后端核心逻辑编写单元测试。
- Agent 指标转换和服务识别编写测试。
- 主要接口执行接口测试。
- MVP 主链路执行集成测试。
- 异常场景记录到测试说明书。
- 答辩前按演示脚本完整彩排。

## 9. 交付风险与应对

| 风险 | 应对 |
| --- | --- |
| P0 任务延期 | 暂停 P2，优先主链路 |
| 数据库配置复杂 | 使用 Docker Compose 固化环境 |
| 真实主机不稳定 | 使用模拟主机兜底 |
| 告警不易触发 | 支持调低阈值和预置告警 |
| 前端页面过多 | 优先主机列表、详情、告警中心 |
| 测试文档不足 | 从开发初期记录测试用例和截图 |

## 10. 当前阶段结论

项目已完成需求梳理和任务拆解，可以进入概要设计阶段。下一阶段应重点产出系统架构图、模块设计、数据库设计、API 分组和数据流设计，为后续编码提供稳定依据。
