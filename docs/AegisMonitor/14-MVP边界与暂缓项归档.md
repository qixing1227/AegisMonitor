# AegisMonitor MVP 边界与暂缓项归档

## 1. 归档结论

本课程设计第一版 MVP 聚焦“一体化监控平台”的可演示主链路：

```text
Agent 采集与上报
-> Spring Boot 后端接收、持久化、告警评估
-> MySQL 保存主机、指标快照、服务清单、告警事件
-> Vue 运维大屏展示主机、指标、服务、告警 ACK
-> 演示数据 seed 支撑 5 台主机规模展示
```

第一版不继续扩展登录、权限、历史曲线、拓扑和规则编辑。它们保留在设计文档和答辩说明中，作为企业级扩展方向，而不是当前冲刺必须完成的代码。

## 2. 已进入 MVP 的能力

| 模块 | 当前交付 |
| --- | --- |
| Agent 纳管 | Agent 注册、心跳、凭证保存、真实主机基础信息采集 |
| 指标采集 | CPU、内存、TCP 连接数等最新快照采集与上报 |
| 服务发现 | Spring Boot、MySQL、Redis、Nginx、Node.js 等服务识别与上报 |
| 后端 API | Agent、指标、服务、告警、ACK、演示数据 seed 接口 |
| 数据持久化 | MySQL 保存 Agent、服务实例、告警事件；测试使用 H2 隔离 |
| 告警闭环 | CPU 高阈值告警生成、去重、列表查询、ACK 和备注 |
| 前端展示 | 主机监控、主机详情、服务组件、告警中心、自动刷新、错误提示 |
| 答辩演示 | 初始化演示数据按钮和固定点击路线 |

## 3. 暂缓项

| 暂缓项 | 当前处理 | 暂缓原因 | 后续扩展方式 |
| --- | --- | --- | --- |
| 登录认证 | 第一版直接进入监控台 | 课程设计重点是监控主链路，不让登录流程占用演示时间 | 增加 Spring Security + JWT 登录 |
| 用户管理 | 只在文档中设计超级管理员、运维工程师、只读研发角色 | 小组演示主机规模有限，真实账号生命周期管理价值不如主链路高 | 增加用户表、角色表和后台用户管理页 |
| 真实 RBAC 权限隔离 | 页面不做真实权限校验 | 当前没有多租户和真实业务部门数据隔离场景 | 在 Controller 层加角色注解，前端按权限隐藏操作 |
| 告警规则编辑 | 使用默认 CPU 高阈值规则 | ACK 闭环比规则编辑更能体现运维业务价值 | 新增规则表、规则编辑页和阈值保存接口 |
| 历史曲线 | 第一版只展示最新指标快照 | 暂不引入 InfluxDB，避免数据库链路复杂度膨胀 | 引入 InfluxDB 或 MySQL 时序表，前端接 ECharts 曲线 |
| 服务拓扑 | 不进入第一版导航 | 当前服务发现能展示服务清单，拓扑需要额外关系建模 | 根据端口、进程和应用配置生成服务依赖图 |
| WebSocket 实时推送 | 使用定时刷新 | 轮询足以支撑课程答辩展示，开发风险更低 | 后端增加事件推送，前端订阅指标和告警变化 |
| 部署运维 | 本地运行和演示文档 | 课程设计主要在实验环境答辩 | 后续补 Docker Compose、Nginx 反向代理和部署手册 |

## 4. 页面导航边界

第一版前端导航只保留已经可工作的入口：

- 主机监控
- 服务组件
- 告警中心

不会在页面中放置登录、用户管理、规则配置、历史曲线、拓扑等不可工作的入口。这样可以避免答辩时出现“看得到但打不开”的半成品界面。

## 5. 答辩话术

可以这样说明范围控制：

> 本项目采用课程设计级 MVP 策略，优先实现监控平台最核心的闭环：Agent 接入、指标采集、服务发现、告警生成、告警确认和大屏展示。登录、权限隔离、规则编辑、历史曲线和拓扑属于企业级扩展能力，已经在设计文档中保留扩展方案，但第一版不进入可运行代码，避免削弱主链路质量。

可以这样说明权限设计：

> 我们在需求分析中设计了超级管理员、运维工程师和只读研发角色。MVP 中主要实现运维工程师视角，因为它覆盖主机监控、服务查看和告警 ACK 这些核心业务动作。后续可在 Spring Security 和前端路由守卫基础上扩展 RBAC。

可以这样说明历史曲线：

> 当前版本展示最新快照，满足课程答辩对实时状态的展示要求。历史曲线需要时序存储和前端图表联动，我们将它作为下一阶段 InfluxDB/ECharts 扩展，避免第一版同时引入过多数据库技术。

## 6. 验收状态

- 第一版暂缓范围已确认并归档。
- 页面导航没有不可工作的功能入口。
- 设计文档中保留了权限、规则、历史曲线、拓扑等扩展方向。
- 答辩话术已明确：课程设计 MVP 已实现主链路，企业级能力作为扩展设计。
