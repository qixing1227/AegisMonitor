# AegisMonitor 项目决策记录

## 1. 项目基本信息

- 项目名称：AegisMonitor 一体化监控平台
- 课题来源：中国电信广东公司实训课题 2026 年版
- 对应课题：课题二，一体化监控平台开发
- 当前阶段：需求澄清阶段收口，进入 PRD 与需求规格整理
- 记录日期：2026-06-04

## 2. 项目定位

本项目定位为课程设计级可运行演示系统，采用生产级思维进行架构设计与文档表达。

系统目标是在有限硬件和课程周期内，实现 Agent 主动采集主机与服务指标、服务端接收与存储、监控台可视化展示、阈值告警、服务拓扑和模拟调用链等核心能力。项目不追求企业生产环境的海量主机、高可用、自动升级、复杂告警收敛和完整分布式追踪能力，但会在设计文档中说明后续演进方向。

## 3. 已确认决策

| 编号 | 决策项 | 最终决策 | 理由 |
| --- | --- | --- | --- |
| D-001 | 项目级别 | 课程设计级可运行演示系统 | 符合 5 人小组、有限主机、课程周期和答辩要求 |
| D-002 | 演示规模 | 2 台真实主机 + 3 台模拟主机 | 贴合课题硬件要求，同时保证答辩现场稳定 |
| D-003 | Agent 平台 | MVP 仅支持 Windows Agent | 小组环境更可控，文档说明可扩展 Linux |
| D-004 | Agent 技术 | Python | 便于使用 psutil 快速采集系统指标 |
| D-005 | 主机指标 | 基础信息、CPU、内存、磁盘、网络、TCP 连接 | 精准覆盖课题要求中的主机信息与主机利用率 |
| D-006 | 服务识别 | Java/Spring Boot、MySQL、Redis、Nginx、Node.js | 覆盖常见企业技术栈，便于演示和文档说明 |
| D-007 | 后端架构 | Spring Boot 模块化单体 | 部署简单，结构清晰，可在文档中说明微服务演进 |
| D-008 | 后端数据访问 | MyBatis-Plus + InfluxDB Java Client | MySQL 业务表使用 MyBatis-Plus，指标数据使用 InfluxDB 客户端 |
| D-009 | 数据库架构 | MySQL + InfluxDB | MySQL 存资产、配置、权限和告警，InfluxDB 存时间序列指标 |
| D-010 | 前端技术 | Vue 3 + Vite + Element Plus + ECharts | 适合企业后台监控台和指标图表展示 |
| D-011 | Agent 通信 | Agent 主动 HTTP/REST 上报 | 实现简单、调试方便、符合课程演示需求 |
| D-012 | 采集频率 | 主机指标 5s，心跳 10s，服务发现 30s，静态信息启动和每 5min | 兼顾演示效果与系统负载 |
| D-013 | 告警能力 | 阈值告警 MVP | 实现 CPU、内存、磁盘、Agent 离线、服务停止等基础告警 |
| D-014 | 调用链 | 服务拓扑 + 模拟调用链 | 回应课题要求，避免完整 OpenTelemetry 追踪的超范围风险 |
| D-015 | 权限模型 | 三角色 RBAC，MVP 实现前两个 | 体现企业级最小权限原则，同时控制实现工作量 |
| D-016 | Agent 纳管 | 注册 Token + 简化审批 | 体现 Agent 接入安全与生命周期管理 |
| D-017 | Agent 配置 | 本地配置文件，后端展示摘要，不做远程下发 | 避免引入配置版本、下发和回滚复杂度 |
| D-018 | 前端刷新 | 定时轮询 + 手动刷新 | 与 5s 采集周期匹配，暂不引入 WebSocket |
| D-019 | 测试策略 | 单元测试、接口测试、集成测试、异常测试 | 满足课程设计测试说明书和工程严谨性要求 |
| D-020 | 项目过程 | 文档化瀑布 + 敏捷迭代实现 | 既满足学校文档要求，也保证功能尽早跑通 |
| D-021 | 部署策略 | Docker Compose 启动 MySQL + InfluxDB，应用本地运行 | 降低数据库部署复杂度，保留本地调试灵活性 |
| D-022 | API 文档 | Swagger UI / OpenAPI | 便于前后端联调、课程文档引用和答辩展示 |

## 4. MVP 明确不做

- 不做生产级高可用集群部署。
- 不做 Agent 自动升级。
- 不做后端远程下发 Agent 配置。
- 不做 WebSocket/SSE 实时推送。
- 不做企业微信、短信、邮件等真实告警通知通道。
- 不做复杂告警收敛、抑制、升级和值班策略。
- 不做完整 OpenTelemetry 分布式追踪。
- 不做 Kubernetes、Docker、云厂商 API 指标采集。
- 不做多租户系统。
- 不做完整研发只读角色的数据隔离实现，仅保留设计说明。
